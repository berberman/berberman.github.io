对内可变对外不可变的 List

berberman 10月 13, 2018
  • 编程
  • 分享到微博
  • 分享到 QQ

前言

在许多场景中都需要有一个内部的容器来储存数据，这个容器对内部是可变的，但可以从外部获取。举给栗子：

class Multicast {
    val connections = mutableListOf<String>()
}

这是一个组播节点，其中持有着其他节点的名字。此时内部实现中需要对 connections 进行增加、移除元素等操作，而外部只能获取这个容器，不能够修改这个容器的内容。

看这愚蠢的高亮，不能识别 val。

Ver. 1

class MulticastA {
    private val _connections = mutableListOf<String>()
    val connections: List<String>
        get() = _connections
}

内部使用 _connections 可变列表进行修改，外部获取到的列表类型是不可变的 List。好的，问题解决了！不过稍等一下，我们思考另一个问题—— 把 connections 强转为 MutableList<String> 会发生什么呢？GG

Ver. 2

class MulticastB {
    private val _connections = mutableListOf<String>()
    val connections
        get() = _connections.toList()
}

既然每次获取到的都是原来可变列表的引用，我们 clone 一下不就好了嘛。如果这样实现，每次外部访问时都会创建一个新的 —— 访问次数多到智熄时，会有 n 个列表被创建出来，我觉得布星。

Ver. 3

class MulticastC {
    var connections = listOf<String>()
        private set
}

可变太辣鸡了，会有各种问题，我们不如。。。

这样虽然解决了问题，内部可以修改变量 connections 的引用，且外部只能访问。然而，每次修改数据都需要创建一个新的 List，这并不是很柯学。

Ver. final

class MulticastD {
    private val _connections = mutableListOf<String>()
    val connections = object : List<String> by _connections {}
}

通过建立匿名实例继承 List<String> 不可变接口，将实现委托给可变列表 _connections，使用不可变列表接口来约定可变列表的行为。这样没有创建许多不必要的实例，也不必使用不可变来解决问题。
